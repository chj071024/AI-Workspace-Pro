# 第一部分：网站定位分析

---

## 1.1 一句话定位

> 我是一名独立开发者，帮助创作者、自由职业者和小企业用 AI + 现代 Web 技术构建实用的数字产品，解决真实业务问题。

---

## 1.2 定位拆解

| 维度 | 内容 |
|------|------|
| **我是谁** | 独立开发者 / AI Automation Developer / Web Developer / Mini Program Developer / AI Agent Builder |
| **我为谁服务** | 个人创作者、自由职业者、小企业、创业团队、本地商户 |
| **我解决什么问题** | 业务效率低、缺少线上存在、需要自动化但不懂技术、需要定制开发但找不到可靠的人 |
| **我怎么解决** | AI 工作流自动化、网站/落地页开发、小程序开发、AI Agent 搭建 |
| **为什么选我** | 关注业务结果而非技术本身、沟通清晰、交付可靠、代码可维护、不过度承诺 |
| **我的差异化** | 不是"写代码的人"，而是"用技术帮客户解决问题的人"；AI + 开发双重能力；独立、灵活、直接沟通 |

---

## 1.3 品牌性格

| 关键词 | 用户感受 |
|--------|----------|
| 专业 | 这个人很靠谱，不是随便找的 |
| 可靠 | 答应的事情会做到 |
| 现代 | 技术栈新、审美在线 |
| 高效 | 不拖沓、沟通直接 |
| 务实 | 不吹牛、不卖焦虑、说实话 |
| 真诚 | 能做就做，不能做会告诉你 |

---

## 1.4 品牌调性

- **语言风格**: 清晰、直接、不绕弯、不用术语吓人、像和一个懂技术的朋友聊天
- **视觉风格**: 极简、高级、克制、大量留白、黑白灰为主、蓝色点缀
- **情感基调**: 信任感 > 兴奋感；专业感 > 炫酷感；亲和力 > 距离感

---

## 1.5 不可妥协的原则

1. **真实性优先**: 所有作品案例必须是真实项目。绝不虚构。
2. **可维护性优先**: 代码结构清晰、模块化，未来可以轻松扩展。
3. **性能优先**: 页面加载速度必须快。Lighthouse 评分目标 95+。
4. **SEO 优先**: 从一开始就构建语义化 HTML 和完整 Meta 信息。
5. **移动端优先体验**: 不能只在桌面端好看，移动端必须同样优质。
6. **不做过度设计**: 不为了炫技加复杂动画。每个元素都有存在的理由。

| **决策标准** | 速度 > 技术能力 > 价格 |
| **如何找到我** | 社交媒体、Indie Hacker 社区、搜索 |
| **关键词** | "能快速交付"、"懂产品"、"不废话" |

---

## 2.2 用户共性需求总结

所有目标用户的共同需求：

1. **信任**: 这是第一关。用户需要相信我是真实、靠谱、有能力的人。
2. **理解**: 用户需要感受到我理解他们的业务和痛点，而不是只想卖服务。
3. **证明**: 用户需要看到真实的作品和案例，证明我有能力交付。
4. **清晰**: 用户需要清楚知道我能做什么、怎么合作、多少钱（大概）。
5. **低门槛**: 联系我的方式必须简单直接，不需要填复杂表单。

---

## 2.3 用户进入网站时的心理状态

| 阶段 | 用户想法 | 网站需要做什么 |
|------|----------|----------------|
| 0秒 | "这是谁？" | 一眼看出身份和价值主张 |
| 5秒 | "他做的是不是我需要的？" | 清晰的服务分类 |
| 15秒 | "他做的怎么样？" | 展示精选作品 |
| 30秒 | "他值得信任吗？" | 关于我、流程、FAQ |
| 60秒 | "怎么联系他？" | CTA按钮、联系方式 |
| 之后 | "我再看看..." | 博客、案例研究、更多作品 |

# 第四部分：页面结构设计

---

## 4.1 首页（Home）— index.html

**设计目标**: 用户在 60 秒内理解我是谁、我做什么、我是否值得信任，并产生联系冲动。

### 模块顺序与设计理由

---

### Hero 区域

**内容**:
- 一句价值主张（核心标语）
- 子标题（补充说明定位）
- 两个 CTA 按钮：「查看我的作品」（主按钮）、「联系我」（次按钮）
- 可选：一张高质量个人照片或极简插画

**设计理由**:
- 用户进入网站的前 3 秒决定了去留
- 价值主张一句话说清楚"我是谁 + 我能帮你做什么"
- 双 CTA 给两种用户路径：想先看看的（浏览作品）和直接想合作的（联系我）
- 克制设计：大量留白、大字号标题、无干扰元素

**参考风格**: Apple 产品页 Hero（大标题 + 极简）、Linear 首页（一句话 + 干净背景）

---

### 价值主张 / 定位声明

**内容**:
- 一段 2-3 句话的简短介绍
- 明确说明服务对象和服务方式
- 强调"解决业务问题"而非"写代码"

**设计理由**:
- Hero 之后的自然延伸
- 让用户确认"这个人做的正是我需要的"
- 用客户能理解的语言，不用术语

---

### 服务概览

**内容**:
- 3-4 个核心服务的卡片式展示
- 每个卡片：图标 + 服务名 + 一句话描述 + 「了解更多」链接
- 服务：AI 工作流自动化 / 网站开发 / 小程序开发 / AI Agent 搭建

**设计理由**:
- 用户需要快速了解"你能具体做什么"
- 卡片式设计方便扫描，不需要读大量文字
- 每个卡片链接到对应的服务详情页
- 不在此处展开所有细节，保持首页简洁

---

### 精选作品

**内容**:
- 展示 3 个最具代表性的项目
- 每个项目：缩略图/示意图 + 项目名 + 简短描述 + 标签（技术/类型）+ 「查看详情」链接
- 「查看全部作品」链接

**设计理由**:
- 作品是最有力的信任证明
- 精选而非全部展示，避免选择疲劳
- 用真实项目建立可信度
- 标签帮助用户快速判断项目类型

---

### 技能展示

**内容**:
- 以简洁方式展示核心技术栈和能力范围
- 可以用极简图标 + 文字，或标签云形式
- 分组：前端 / 后端 / AI / 工具 / 平台

**设计理由**:
- 快速传达技术广度和专业度
- 让不同类型的客户看到自己关心的那部分
- 不过度展示技术细节，保持可读性

---

### 开发流程 / 合作流程

**内容**:
- 4-5 步的极简流程展示
- 例如：沟通需求 → 方案确认 → 开发执行 → 交付上线 → 持续支持
- 每一步有简短说明

**设计理由**:
- 降低用户的心理门槛：让他们知道"合作会是什么样"
- 专业感：有流程 = 有经验 = 靠谱
- 消除不确定性带来的顾虑

---

### 为什么选择我

**内容**:
- 3-4 个差异化价值点
- 例如：关注业务结果 / 直接沟通无中间商 / 代码可维护不挖坑 / AI 加持更高效率
- 每个点配简短解释

**设计理由**:
- 在用户即将滚动到页面底部时给出最终"推动"
- 直接回应潜在客户最常见的顾虑
- 差异化竞争：不是在比谁便宜，而是在比谁更靠谱

---

### FAQ

**内容**:
- 5-6 个最常见问题 + 简短回答
- 例如：合作流程是什么？怎么收费？多久能交付？不满意怎么办？能后期维护吗？技术栈是什么？

**设计理由**:
- 在用户主动联系前消除最后疑虑
- 减少低质量咨询（用户问的问题都在 FAQ 里回答了）
- FAQ 对 SEO 也有利

---

### CTA 区域（最终行动号召）

**内容**:
- 大标题（如"准备好开始合作了吗？"）
- 简短引导文案
- 主 CTA 按钮（「立即联系我」）
- 可选：邮箱、微信等联系方式

**设计理由**:
- 用户在页面底部需要一个明确的下一步行动
- 不要假设用户会主动去找联系方式——放到他们面前
- 语气友好、不施压

---

### Footer

**内容**:
- 网站导航链接（首页 / 关于 / 服务 / 作品 / 博客 / 联系）
- 服务列表链接
- 社交媒体链接
- 版权信息
- 「回到顶部」按钮

**设计理由**:
- 给用户最终一个快速导航的入口
- 社交媒体链接提供更多信任维度
- 简洁、不臃肿、不过度设计

---

## 4.2 关于我（About）— about.html

**设计目标**: 让用户觉得"这是个真实、靠谱、有想法的人"，建立个人层面的信任。

### 模块顺序与设计理由

---

### 个人介绍

**内容**:
- 一张高质量个人照片（可选）
- 一段真诚的个人介绍（非简历式）
- 说明：我是谁、我做什么、为什么做这个、我的理念

**设计理由**:
- 关于页面是"建立人际信任"的关键页面
- 不要像简历那样罗列经历，而是讲故事
- 让用户感受到背后是一个真实的人

---

### 技能与工具

**内容**:
- 技术栈展示（比首页更详细但不冗长）
- 工具和工作流偏好
- 持续学习的方向

**设计理由**:
- 证明专业能力有据可查
- 让潜在客户评估技术匹配度
- 展示"持续学习"的态度

---

### 工作理念

**内容**:
- 3-5 条核心理念
- 例如：关注业务结果 / 代码可维护 / 真诚沟通 / 不接做不了的项目
- 每条有简短解释

**设计理由**:
- 让客户了解我的工作方式和价值观
- 筛选客户：认同这些理念的客户更容易合作
- 建立差异化：不是所有开发者都有清晰的工作理念

---

### 个人故事（可选）

**内容**:
- 简短的职业经历或转折点
- 为什么选择独立开发这条路
- 对未来的规划

**设计理由**:
- 故事比简历更有记忆点
- 增加真实性和亲和力
- 为品牌增加厚度

---

## 4.3 服务（Services）— services.html

**设计目标**: 让用户清楚知道我提供什么服务、怎么交付、为什么值得付费。

### 模块顺序与设计理由

---

### 页面标题

**内容**:
- 标题 + 简短描述

---

### 服务列表（详细版）

**每个服务包含**:
- 图标
- 服务名称
- 详细描述（比首页更展开）
- 我能帮你做什么（具体场景举例）
- 技术栈 / 平台
- 交付物说明
- 「咨询此服务」CTA 按钮

**服务清单**:

1. **AI 工作流自动化**
   - 场景：AI 客服、内容生成、数据自动处理、工作流自动化
   - 价值：节省时间、减少人工重复劳动、提高效率

2. **网站开发**
   - 场景：个人网站、企业官网、落地页、产品展示站
   - 价值：专业线上形象、SEO、24/7 展示

3. **小程序开发**
   - 场景：预约系统、内部管理工具、客户互动工具
   - 价值：微信生态获客、轻量级应用、无需下载

4. **AI Agent 搭建**
   - 场景：智能客服 Agent、自动化运营 Agent、数据分析 Agent
   - 价值：7×24 自动运行、减少人力成本、规模化服务

**设计理由**:
- 每个服务详细展开，帮助用户判断"这个服务是否适合我"
- 用具体场景举例，让非技术用户也能理解
- 每个服务都有独立 CTA，降低行动门槛

---

### 服务流程

**内容**:
- 统一的合作流程（5步）
- 每步说明我做什么、客户需要做什么
- 时间预估

**设计理由**:
- 统一的服务流程适用于所有服务类型
- 让用户对"合作起来是什么样"有清晰预期
- 时间预估帮助用户做决策

---

### 服务 FAQ

**内容**:
- 针对服务的常见问题
- 如：报价范围、交付周期、售后支持、修改次数

**设计理由**:
- 与服务页面的 FAQ 可以更有针对性
- 减少因信息不明确而放弃咨询的情况

---

### CTA

**内容**:
- 引导用户联系或咨询

---

## 4.4 作品（Projects）— projects.html + 详情页

**设计目标**: 用真实作品证明能力，让用户产生"这个人的作品不错，我想找他做类似的项目"的想法。

### 模块顺序与设计理由

---

### 页面标题

**内容**:
- 标题 + 简短描述

---

### 筛选器

**内容**:
- 按类型筛选：全部 / 网站 / 小程序 / AI自动化 / 其他
- 简单按钮切换，不复杂

**设计理由**:
- 帮助用户快速找到相关案例
- 不同客户关心不同类型的作品
- 简单筛选就够，不用复杂交互

---

### 作品列表

**每个作品卡片**:
- 缩略图或预览图
- 项目名称
- 简短描述（1-2句话）
- 标签（技术栈 / 类型）
- 「查看详情」链接

**设计理由**:
- 网格或列表布局，清晰可扫描
- 每个卡片提供足够信息帮助用户决定是否点击
- 图片是关键：真实截图 > 花哨 mockup

---

### 作品详情页（每个作品独立页面）

**内容**:
- 项目名称
- 项目背景（客户是谁、面临什么问题）
- 解决方案（我做了什么、怎么做的）
- 技术栈
- 成果/效果（如果有数据更好）
- 截图或预览
- 客户评价（如果有）
- 「有类似需求？联系我」CTA

**设计理由**:
- 详情页是深度信任建立的关键
- 展示完整思考过程，而不只是最终截图
- "问题 → 解决方案 → 成果"的叙事结构更有说服力
- 每个详情页的 CTA 转化率最高

---

## 4.5 博客（Blog）— blog.html + 文章页

**设计目标**: SEO 获客 + 展示专业度 + 建立长期信任。

### 模块顺序与设计理由

---

### 文章列表

**内容**:
- 文章标题
- 发布日期
- 分类/标签
- 简短摘要
- 阅读时间预估

**设计理由**:
- 清晰的信息层级
- 帮助用户快速浏览和选择
- 标签和分类便于筛选

---

### 文章详情页

**内容**:
- 文章标题
- 发布日期 + 阅读时间
- 文章正文（结构化、有目录）
- 标签
- 上一篇/下一篇导航
- 相关文章推荐
- CTA（「有相关需求？联系我」）

**设计理由**:
- 文章质量 > 文章数量
- 结构化内容（h2/h3 标题层级）对 SEO 友好
- 相关文章推荐增加页面停留时间
- 每篇文章底部放 CTA，利用阅读后的信任转化

---

### 博客侧边栏/底部

**内容**:
- 分类列表
- 最新文章
- CTA

## 4.6 联系（Contact）— contact.html

**设计目标**: 让用户尽可能方便地联系到我，同时收集必要的初步信息。

### 模块顺序与设计理由

---

### 联系方式

**内容**:
- 邮箱地址
- 微信（可选展示）
- 响应时间承诺
- 社交媒体链接

**设计理由**:
- 直接展示联系方式降低心理门槛
- 响应时间承诺建立信任（"通常在24小时内回复"）

---

### 联系表单

**字段**:
- 姓名（必填）
- 邮箱（必填）
- 服务类型（下拉选择）
- 项目描述（必填，textarea）
- 预算范围（可选）

**设计理由**:
- 最小化必填字段，降低填写门槛
- 服务类型帮助快速分类
- 项目描述让第一次沟通更有准备
- 不要求电话号码（尊重隐私、降低门槛）

---

### 联系页 FAQ

**内容**:
- 联系方式相关 FAQ
- 如：怎么收费、能免费咨询吗、需要准备什么

## 5.1 主要用户旅程

```
Google 搜索 "个人网站开发"
        ↓
    搜索结果点击进入（可能是博客文章或首页）
        ↓
    首页 Hero → 确认"这个人做网站开发"
        ↓
    查看精选作品 → "作品不错，风格我喜欢"
        ↓
    查看服务页 → "网站开发服务，正是我需要的"
        ↓
    查看开发流程 → "流程清晰，靠谱"
        ↓
    点击「联系我」→ 填写表单或直接发邮件
        ↓
    转化 ✓
```

---

### 旅程 B：浏览型用户（社交媒体 / 推荐进入）

```
朋友推荐 / 社交媒体链接
        ↓
    首页 Hero → "先看看这是什么"
        ↓
    滚动浏览 → 看服务概览、作品
        ↓
    查看关于页 → "先了解这个人"
        ↓
    查看作品详情 → "这个项目有意思"
        ↓
    收藏 / 记住 → 有需求时会回来
        ↓
    可能不会立即转化，但建立了品牌认知 ✓
```

---

### 旅程 C：对比型用户（在多个开发者中比较）

```
首页 Hero → "看起来挺专业的"
        ↓
    直接看作品页 → "看看作品质量"
        ↓
    看作品详情 → "分析技术水平和审美"
        ↓
    看关于页 → "这个人值不值得信任"
        ↓
    看 FAQ → "确认顾虑"
        ↓
    联系 → "应该比其他人靠谱"
        ↓
    转化 ✓
```

---

### 旅程 D：重复访问用户（老访客 / 潜在回头客）

```
直接进入 /blog → 看新文章
        ↓
    或进入首页 → 看有没有新作品
        ↓
    发现新内容 → 信任加深
        ↓
    某天有需求 → 直接联系
        ↓
    转化 ✓
```

# 第六部分：网站未来扩展路线

---

## 6.1 短期扩展（V1.1 - V1.5）

### 案例研究（Case Studies）
- **内容**: 每个项目深度案例页面（背景 → 挑战 → 方案 → 成果 → 客户评价）
- **触发条件**: 完成 2-3 个有代表性的项目后
- **架构准备**: projects/ 目录已预留，详情页模板可复用

### 客户评价墙（Testimonials）
- **内容**: 汇集客户评价文字/截图
- **触发条件**: 积累 3+ 客户评价后
- **架构准备**: 可在首页增加 Testimonials Section，或在关于页增加

### 定价页（Pricing）
- **内容**: 服务价格范围或套餐
- **触发条件**: 服务标准化程度足够高时
- **架构准备**: 新增 pricing.html

---

## 6.2 中期扩展（V2.0）

### AI 产品展示（AI Products）
- **内容**: 自研的 AI 工具 / Agent 产品展示页
- **触发条件**: 有可公开的 AI 产品时
- **架构准备**: products/ 目录预留

### 工具集（Tools）
- **内容**: 免费在线工具集合（如计算器、生成器、转换器等）
- **目的**: SEO 获客 + 展示技术能力
- **架构准备**: tools/ 目录预留

### 资源库（Resources）
- **内容**: 推荐工具清单、模板下载、教程文章
- **目的**: 增加回访 + SEO + 建立社区
- **架构准备**: resources/ 目录预留

---

## 6.3 长期扩展（V3.0+）

### 课程（Courses）
- **内容**: 录播课程 / 直播课程 / Workshop
- **目的**: 新的收入来源
- **架构准备**: courses/ 目录预留

### 多语言（i18n）
- **内容**: 英文版 / 其他语言版本
- **触发条件**: 有海外客户需求时
- **架构准备**: 使用语义化 HTML + 可配置的语言切换

### CMS 集成
- **内容**: 博客和作品的后台管理系统
- **触发条件**: 内容量大到手动管理困难时
- **架构准备**: 可以先使用静态 JSON 数据驱动，后续平滑迁移到 CMS

### 在线预约（Booking）
- **内容**: Calendly 集成或自建预约系统
- **触发条件**: 咨询量大到需要自动排期时
- **架构准备**: 初期使用 Calendly 嵌入即可

---

## 6.4 架构的可扩展性设计

为了保证未来的扩展不需要重构，V1 就做好以下准备：

1. **模块化 CSS**: 每个组件独立 CSS 文件，不互相依赖
2. **组件化 JS**: 每个交互模块独立 JS 文件
3. **数据驱动**: 作品、服务等重复内容用 JSON 数据渲染，而非硬编码 HTML
4. **语义化 HTML**: 使用 header、main、nav、article、section、footer 等标签
5. **URL 结构清晰**: 一致的、可预测的 URL 结构
6. **目录结构预留**: 未来可能用到的目录提前创建

## 7.2 CSS 文件组织原则

```
css/
├── reset.css         # 第1层：重置浏览器默认样式
├── variables.css     # 第2层：CSS 变量定义
├── global.css        # 第3层：全局基础样式
```

# 第八部分：开发路线（Milestones）

---

## Milestone 1：基础架构搭建（预计 1-2 天）

### 目标
搭建项目骨架，确保开发环境、目录结构、基础样式就绪。

### 具体任务

1. **创建完整目录结构**
   - 按照第七部分的目录结构创建所有文件夹和空文件
   - 确保 `.gitignore` 就位

2. **CSS 基础体系**
   - `reset.css`: 使用 modern CSS reset
   - `variables.css`: 定义颜色、间距、字体、阴影、圆角等 Token
   - `global.css`: body、标题、段落、链接、列表、图片等基础样式
   - `layout.css`: 页面最大宽度、网格系统、section 间距

3. **HTML 模板基础**
   - 创建所有页面的 HTML 骨架文件
   - 每个页面包含：正确的 DOCTYPE、lang、charset、viewport meta
   - 所有页面共享的 `<head>` 部分抽取为统一模板

4. **SEO 基础设施**
   - 定义全局 Meta 模板（title、description、keywords）
   - 创建 `robots.txt`
   - 创建 `sitemap.xml`（初期手动，后续可自动化）
   - Open Graph + Twitter Card 标签模板
   - Schema.org JSON-LD 模板（Person + WebSite）

5. **全局 Header + Footer 组件**
   - Header: Logo、导航、移动端汉堡菜单
   - Footer: 导航链接、社交媒体、版权
   - 所有页面引用相同的 Header 和 Footer（使用 JS 动态加载或手动复制）

6. **全局 JS 基础**
   - `main.js`: 初始化
   - `header.js`: 导航滚动效果、移动端菜单切换
   - `animation.js`: Intersection Observer 滚动入场动画（极简）

7. **响应式断点定义**
   - Mobile: < 640px
   - Tablet: 640px - 1024px
   - Desktop: > 1024px
   - Large Desktop: > 1280px

### 交付物
- 完整的项目目录
- 所有页面 HTML 骨架
- 完整 CSS 基础体系
- 可用的 Header 和 Footer
- SEO 基础设施文件

---

## Milestone 2：首页开发（预计 2-3 天）

### 目标
完成首页全部模块的开发，首页是网站最重要的页面。

### 具体任务

1. **Hero 区域**
   - 大标题（价值主张文案）
   - 子标题
   - 双 CTA 按钮
   - 极简背景（纯色或 subtle 渐变）

2. **服务概览区域**
   - 4 张服务卡片
   - 图标 + 标题 + 描述 + 链接

3. **精选作品区域**
   - 3 张作品卡片或横向展示
   - 缩略图 / 示意图

4. **技能展示区域**
   - 技术栈极简展示
   - 标签/图标式呈现

5. **开发流程区域**
   - 5 步流程
   - 数字序号 + 简短文字

6. **为什么选择我区域**
   - 3-4 个价值点

7. **FAQ 区域**
   - 折叠面板（Accordion 交互）
   - 5-6 个问题

8. **CTA 区域**
   - 大标题 + CTA 按钮

9. **响应式适配**

# 附录 C：技术决策记录

---

## C.1 为什么 V1 选择纯 HTML/CSS/JS

| 原因 | 说明 |
|------|------|
| **学习成本** | 这是个人网站，我自己维护。原生技术不需要学习框架 |
| **性能** | 零依赖 = 极小体积 = 极致加载速度 |
| **可维护性** | 代码完全可控，没有框架版本升级的烦恼 |
| **部署简单** | 静态文件，任何服务器甚至 GitHub Pages 都能部署 |
| **未来可升级** | 结构清晰，以后迁移到 React/Vue 也很容易 |

---

## C.2 为什么不用 Tailwind

| 原因 | 说明 |
|------|------|
| **可读性** | 原生 CSS（特别是 BEM 命名）比满屏 utility class 更易读 |
| **学习曲线** | 未来可能有合作者不想学 Tailwind |
| **完全控制** | CSS 自定义属性体系足够强大，不需要额外的抽象层 |
| **迁移友好** | 以后升级到任何框架，CSS 都可以直接复用 |

---

## C.3 为什么不一开始就用 React/Vue

| 原因 | 说明 |
|------|------|
| **过度设计** | 个人展示型网站不需要 SPA |
| **SEO** | 静态 HTML 对搜索引擎最友好，不需要 SSR |
| **内容更新频率** | 作品和博客更新不频繁，手动编辑即可 |
| **KISS 原则** | 能用简单方案解决的问题不需要复杂方案 |
